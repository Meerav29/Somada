# Somada — Agent Context

**Somada** is a personal Apple Health dashboard for Meerav Shah (Penn State student). It visualizes 8+ months of wearable data (steps, sleep, HRV, heart rate) and lets you chat with an AI analyst about the data. The app is deployed on **Vercel** and uses **Supabase** for auth and data storage.

---

## File Structure

```
health-dashboard/
├── index.html          # Entire frontend — single HTML file, vanilla JS, no build step
├── server.py           # Local-dev-only HTTP server (NOT used on Vercel)
├── parse_health.py     # Local-dev-only XML parser CLI (NOT used on Vercel)
├── vercel.json         # Vercel config: rewrites all non-api/* paths to index.html
├── .env                # Local env vars (gitignored)
└── api/                # Vercel serverless functions (Python, one file = one route)
    ├── health.py       # GET  /api/health      — returns parsed health JSON
    ├── chat.py         # POST /api/chat        — proxies Claude/Gemini with health context
    ├── config.py       # GET  /api/config      — returns which AI providers are configured
    └── supabase_config.py  # GET /api/supabase_config — returns Supabase URL + anon key to frontend
```

## Deployment: Two Modes

### Vercel (production)
- `api/*.py` files are Vercel Python serverless functions. Each file exports a class named `handler` that subclasses `BaseHTTPRequestHandler`.
- `vercel.json` rewrites all non-`/api/*` requests to `index.html`.
- The filesystem is **ephemeral** — no files written at runtime persist. All data lives in Supabase.
- No `api/upload.py` exists or is needed — upload/parsing happens in the browser.

### Local dev (`python server.py`)
- `server.py` is a `ThreadedHTTPServer` that handles all routes itself (including `/api/*`).
- It reads health data from `health_data.json` (generated by `parse_health.py`).
- Workflow: `python parse_health.py export.xml` → generates `health_data.json` → `python server.py`.
- **Do not modify `server.py` for Vercel-related changes** — it only affects local dev.

## Supabase Setup

### Auth
- Supabase email/password auth. The frontend fetches credentials from `/api/supabase_config`, creates a `supabaseClient` via `window.supabase.createClient(url, anonKey)`.
- `initSupabase()` (index.html) runs on load: checks for existing session, shows auth overlay if not logged in.
- `currentUser` (global) is `null` when logged out, populated with the Supabase user object when logged in.
- The auth overlay blocks the app until sign-in succeeds.

### Database
One table: **`health_data`**

```sql
CREATE TABLE health_data (
  id integer PRIMARY KEY DEFAULT 1,   -- always 1 row (one user)
  data jsonb NOT NULL,                -- the full parsed health JSON
  updated_at timestamptz DEFAULT now()
);
ALTER TABLE health_data ENABLE ROW LEVEL SECURITY;
CREATE POLICY "allow_public_read"  ON health_data FOR SELECT USING (true);
CREATE POLICY "allow_auth_write"   ON health_data FOR ALL    USING (auth.role() = 'authenticated');
```

- Always one row (`id = 1`). Uploading new data does an upsert on `id`.
- Reads are public (anon key works) — `/api/health` and `/api/chat` use the anon key server-side.
- Writes require auth — only the browser (with a valid JWT session) can upsert.

## Upload Flow (Vercel)

The upload runs entirely in the browser — no server involvement for parsing:

1. User selects `export.xml` on the upload page
2. `handleFileUpload()` validates the file (must be `.xml`, non-empty, < 600 MB)
3. **`parseHealthXML(file, onProgress)`** reads the file with `file.text()` and iterates `<Record>` tags using regex, aggregating into daily buckets — a JavaScript port of `parse_health.py`
4. Progress bar updates every 5,000 records via `await new Promise(r => setTimeout(r, 0))` yield
5. Result is filtered to last 8 months and summary stats are computed
6. `supabaseClient.from('health_data').upsert({id: 1, data: result}, {onConflict: 'id'})` writes to Supabase
7. `healthData` global is set immediately; user is navigated to dashboard

**Auth guard:** `handleFileUpload` checks `!currentUser` before the upsert (not `!supabaseClient` — the client object exists even when logged out).

### Parser constants (index.html, near top of `<script>`)
- `HEALTH_METRIC_MAP` — maps Apple Health `HKQuantityTypeIdentifier*` strings to short field names
- `SLEEP_ASLEEP` — Set of `HKCategoryValueSleepAnalysis*` values that count as "asleep" (not in-bed/awake)
- `LIFE_EVENTS` — Hardcoded academic calendar events (Finals, breaks, semesters). **Keep in sync with `parse_health.py`'s `LIFE_EVENTS`** — they are identical arrays maintained in two places.

## API Endpoints

All `api/*.py` files follow the same pattern: a class `handler(BaseHTTPRequestHandler)` with `do_GET`/`do_POST` and a `_send_json()` helper. No frameworks.

### `GET /api/health`
Returns the full health JSON. Tries Supabase first (if `SUPABASE_URL`/`SUPABASE_ANON_KEY` are set), falls back to local `health_data.json`.

### `POST /api/chat`
Body: `{ message, history, provider }`. Loads health data the same way as `/api/health`, builds a system prompt from the data, and proxies to Claude (`claude-sonnet-4-20250514`) or Gemini (`gemini-2.0-flash`). Returns `{ reply, provider }`.

### `GET /api/config`
Returns `{ providers: { claude: bool, gemini: bool } }` — whether each API key is set.

### `GET /api/supabase_config`
Returns `{ url, anonKey }` from env vars. The frontend uses this to initialise the Supabase JS client.

## Frontend Architecture (index.html)

Single-file vanilla JS app (~1750 lines). No framework, no build step.

### Key globals
| Variable | Type | Purpose |
|---|---|---|
| `healthData` | object \| null | Parsed health JSON; null until loaded |
| `supabaseClient` | object \| null | Supabase JS client; null if not configured |
| `currentUser` | object \| null | Supabase user; null when logged out |
| `currentPage` | string | Active page name |
| `currentProvider` | string | `'claude'` or `'gemini'` |
| `charts` | object | Chart.js instances keyed by canvas ID |

### Dashboard — Bento Grid
The default `dashboard` page uses a **bento grid** layout (4-column, 3-row CSS grid). Cards:
- `.bento-steps` — steps count + mini sparkline
- `.bento-sleep` — sleep hours
- `.bento-hrv` — HRV
- `.bento-hr` — heart rate
- `.bento-spo2` — SpO₂
- `.bento-act` — active calories
- `.bento-trend` — multi-metric trend chart (full-width bottom row)

Each bento card has a coloured background (`--card-steps`, `--card-sleep`, etc.), a hero number, delta badge, and optional sub-label. Cards animate on hover (`translateY(-2px)`).

### Navigation
`navigate(page)` sets `currentPage` and calls `renderPage(page)`. Hash-based: `window.location.hash` reflects current page. Pages: `dashboard`, `sleep`, `activity`, `recovery`, `insights`, `chat`, `settings`, `upload`.

### Data access helpers
```js
daily(days)              // last N days from healthData.daily as array
getDailyArray(metric, days)  // { labels: dates[], values: metric[] }
avg(arr)                 // mean of non-null values
```

### Startup sequence
```
DOMContentLoaded
  └─ initSupabase()
       ├─ (no config) → loadAppData() + showApp()    // Supabase not configured
       ├─ (session exists) → loadAppData() + showApp() // already logged in
       └─ (no session) → showAuthOverlay()            // wait for login
                              └─ SIGNED_IN event → loadAppData() + showApp()
```

`loadAppData()` fetches `/api/health` and `/api/config` in parallel and populates `healthData` and `availableProviders`.

Chart.js 4.4.1 (CDN). All charts are stored in the `charts` global object. When re-rendering a page, existing charts are destroyed before creating new ones. Theme-aware: chart colors use CSS variable values read at render time.

## Common Pitfalls

**`supabaseClient` vs `currentUser` for auth guards**
`supabaseClient` is always non-null after `initSupabase()` runs — it's just the JS client object. It exists whether the user is logged in or not. Always check `currentUser` to test if a user is actually authenticated. Using `!supabaseClient` as an auth guard silently passes and the request goes through with the anon role, which RLS then blocks with a cryptic "row-level security policy" error.

**`LIFE_EVENTS` is duplicated**
The academic calendar events array exists in two places: `parse_health.py` (line 14, used for local dev parsing) and `index.html` (`LIFE_EVENTS` const, used by the browser parser). They must stay in sync. If you add or change an event in one, update the other.

**Do not add `/api/upload.py` or `/api/reparse.py`**
These routes do not exist on Vercel and are not needed. Parsing is done in the browser by `parseHealthXML()` in `index.html`. Adding a server-side upload handler won't work anyway due to Vercel's 4.5 MB request body limit and 10-second function timeout — Apple Health exports are commonly 100–500 MB.

**`server.py` is local-dev only**
Changes to `server.py` have no effect on the Vercel deployment. If an `/api/*` route is broken on Vercel, fix the corresponding `api/*.py` file. The two codepaths are completely independent.

**`api/chat.py` and `api/health.py` both load health data independently**
Both files have their own `load_health_data()` / `fetch_from_supabase()` logic. If you change how health data is fetched (e.g. table name, column name, query params), update both files.

**Chart.js instances must be destroyed before re-render**
If a canvas already has a Chart.js instance attached and you create a new one on the same canvas, Chart.js throws an error. The render functions handle this via `charts[id]?.destroy()` before creating a new chart. Don't skip this when adding new chart pages.

**`daily` in `healthData` is a dict keyed by date string, not an array**
`healthData.daily` is `{ "2025-01-15": {...}, ... }`. The helper `daily(n)` converts it to a sorted array of the last N days. Always use the helper or `Object.values(healthData.daily)` — never assume it's already an array.
